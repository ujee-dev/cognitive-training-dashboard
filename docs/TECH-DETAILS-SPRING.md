# 🔵 Spring Boot 기술 명세서 (Backend Migration)

본 문서는 **NestJS (MongoDB)** 기반으로 구축된 인지 훈련 성과 분석 시스템을
**Spring Boot 3 (Java 17) + PostgreSQL** 환경으로 마이그레이션하면서 적용한
**아키텍처 설계와 주요 기술 구현 과정**을 정리한 기술 문서입니다.

기존 시스템의 데이터 구조와 분석 로직을 기반으로 관계형 데이터베이스 구조로 재설계하여
**데이터 무결성, 타입 안정성, 유지보수성 개선**을 목표로 했습니다.

---

> **Legacy 시스템 설계 문서**

**상세 설계, 점수 계산 방식, 화면, 성능 최적화(lighthouse)** 등의
서비스 기능 설계에 대한 상세 내용은 아래 Legacy 기술 문서를 참고할 수 있습니다.

해당 문서는 **본 Spring Boot 마이그레이션 문서의 기반 시스템 설계 참고 자료**입니다.

# 1. 마이그레이션 결정 배경 (Migration Strategy)

## 1.1 기술 스택 전환 이유

### 데이터 무결성 확보

기존 MongoDB 기반 구조에서는 데이터 간 관계가 애플리케이션 로직에 의존하는 형태였습니다.

이를 PostgreSQL 기반 **관계형 데이터 모델**로 재설계하고
**Foreign Key 제약 조건과 트랜잭션(ACID)**을 적용하여 데이터 정합성을 강화했습니다.

---

### 타입 안정성 강화

Java 17의 정적 타입 시스템과 JPA 매핑을 활용하여 컴파일 단계에서 타입 검증이 가능하도록 구성했습니다.

이를 통해 서비스 로직에서 발생할 수 있는 일부 런타임 오류 가능성을 줄이고 코드 안정성을 개선했습니다.

---

### 계층형 아키텍처 적용

Spring Boot의 **Layered Architecture** 구조를 적용하여 시스템을 다음과 같이 분리했습니다.

| 계층       | 역할                  |
| ---------- | --------------------- |
| Controller | API 요청 및 응답 처리 |
| Service    | 비즈니스 로직 처리    |
| Repository | 데이터 접근 계층      |

이 구조를 통해 **비즈니스 로직과 데이터 접근 책임을 명확히 분리**하고 유지보수성을 높였습니다.

---

# 2. 데이터 아키텍처 & 스키마 설계 (PostgreSQL)

MongoDB의 문서 기반 구조를 관계형 스키마로 재설계하여 **데이터 간 참조 관계와 무결성 제약을 명확하게 정의**했습니다.

| Entity     | 설명                | 주요 특징                                            |
| ---------- | ------------------- | ---------------------------------------------------- |
| User       | 사용자 계정 정보    | BCrypt 기반 비밀번호 암호화, Refresh Token 해시 저장 |
| Game       | 훈련 게임 메타 정보 | 게임 식별 코드 및 활성화 상태 관리                   |
| GameConfig | 난이도 설정 데이터  | EASY / NORMAL / HARD 난이도 가중치 관리              |
| GameRecord | 사용자 플레이 기록  | 랭킹 및 통계 조회를 위한 인덱스 적용                 |

특히 `GameRecord` 테이블은 **랭킹 조회 및 통계 계산 성능을 고려하여 인덱스를 구성**했습니다.

# 3. 핵심 기술 구현 (Technical Implementation)

## 3.1 MongoDB Aggregate → JPQL 기반 분석 로직 이식

기존 NestJS 환경에서는 MongoDB **Aggregate Pipeline**을 이용하여 성과 데이터를 계산했습니다.

Spring Boot 마이그레이션 과정에서는 이를 **JPQL 기반 조회와 DTO Projection 방식**으로 재구성했습니다.

---

### DTO Projection 적용

대시보드 및 통계 API에서 필요한 데이터만 조회하기 위해 **DTO Projection 방식**을 적용했습니다.

이를 통해 다음과 같은 효과를 얻을 수 있었습니다.

- 불필요한 엔티티 로딩 방지
- 조회 데이터 크기 감소
- API 응답 처리 효율 개선

---

### 랭킹 계산 로직 구현

사용자 랭킹은 다음 과정을 통해 계산됩니다.

1. 사용자별 최고 점수(MAX) 조회
2. 동일 점수 사용자 그룹화
3. 동일 점수 사용자에게 동일 순위 부여

이 과정은 **서비스 계층에서 랭킹 계산 로직을 구현**하여 랭킹 정책을 코드 레벨에서 관리하도록 구성했습니다.

---

> ℹ️ **분석 로직 상세 문서**

게임 점수 계산식, 랭킹 산정 방식, 통계 계산 알고리즘 등
서비스 분석 로직에 대한 상세 설계 내용은 Legacy NestJS 시스템 문서에서 확인할 수 있습니다.

- **Legacy Backend**: [NestJS 기술 명세서](./TECH-DETAILS-NEST.md)

## 3.2 Spring Security 기반 인증 구조

기존 NestJS Passport 기반 인증 구조를 **Spring Security 기반 인증 구조로 재구성**했습니다.

주요 구성 요소:

- JWT 기반 인증
- Refresh Token Rotation
- Stateless 인증 구조

---

### Refresh Token Rotation

토큰 재발급 시 DB에 저장된 Refresh Token 해시 값을 갱신하는 방식으로 구성하여
**탈취된 토큰의 재사용 가능성을 줄이는 인증 구조**를 적용했습니다.

또한 사용자 상태 변경(비밀번호 변경, 계정 비활성화 등)이 발생할 경우
인증 상태를 관리할 수 있도록 설계했습니다.

# 4. 주요 트러블슈팅 (Troubleshooting)

## 4.1 JPA 지연 로딩(Lazy Loading)과 JSON 직렬화 문제

### 문제

엔티티 간 연관 관계가 있는 상태에서 API 응답으로 엔티티를 직접 반환할 경우
Hibernate Proxy 객체가 Jackson 직렬화 과정에서 처리되지 않아 다음 오류가 발생했습니다.

```
Type definition error
```

---

### 해결

API 응답 전용 **DTO(Data Transfer Object) 패턴**을 적용하여 다음 구조로 변경했습니다.

```
Entity → Service → DTO → Controller → Response
```

이 방식으로 다음 문제를 해결했습니다.

- Lazy Loading Proxy 직렬화 문제 방지
- 순환 참조 문제 예방
- API 응답 구조 명확화

---

## 4.2 Multipart 데이터 처리 로직 개선

### 문제

프로필 이미지 수정 API에서 다음 세 가지 상황을 동시에 처리해야 했습니다.

1. 새로운 이미지 업로드
2. 이미지 삭제 요청
3. 기존 이미지 유지

Spring 기본 바인딩 과정에서 빈 문자열(`""`) 처리와 파일 파라미터 처리 간 충돌이 발생했습니다.

---

### 해결

`MultipartHttpServletRequest`를 사용하여 요청 데이터를 직접 처리하도록 구조를 변경했습니다.

처리 우선순리를 다음과 같이 정의했습니다.

```
1. 파일 업로드
2. 삭제 요청("")
3. 기존 데이터 유지
```

이를 통해 프로필 이미지 업로드 및 삭제 동작을 안정적으로 처리할 수 있도록 구현했습니다.

---

## 4.3 Hibernate ddl-auto 설정 관리

### 문제

초기 개발 과정에서 `ddl-auto=create` 설정으로 인해 애플리케이션 실행 시 데이터가 초기화될 가능성이 확인되었습니다.

---

### 해결

환경별 설정을 다음과 같이 분리했습니다.

| 환경                | ddl-auto |
| ------------------- | -------- |
| 개발 환경           | update   |
| 데이터 안정 필요 시 | none     |

이를 통해 개발 편의성과 데이터 안정성을 모두 고려한 설정 전략을 적용했습니다.

# 5. 성능 및 결과

### API 응답 처리 개선

Spring Boot 기반 구조로 마이그레이션 이후
통계 및 랭킹 조회 API의 데이터 조회 구조가 단순화되면서 **API 응답 처리 과정이 안정화되었습니다.**

---

### 코드 안정성 개선

Java의 정적 타입 시스템과 JPA 매핑을 활용하여
컴파일 단계에서 타입 검증이 가능하도록 구성함으로써 코드 안정성과 유지보수성을 개선했습니다.
